# 设计评审 (PPQA + DAR)

## 用途
合规检查、争议决策。

## 触发时机
设计完成，准备输出设计文档前。

## 输入
- 需求理解纪要
- 信息架构文档
- 交互设计方案
- 视觉设计规范
- 组件规范文档

## 输出
```markdown
## 设计评审报告

### 合规性检查
| 检查项 | 状态 | 备注 |
|--------|------|------|
| 使用已有组件库 | ✅ | Element Plus |
| 使用设计Token | ✅ | 颜色/字体/间距全部复用 |
| 无新视觉语言 | ✅ | 未引入新样式 |
| 交互模式一致 | ✅ | 符合现有规范 |
| 可实现性 | ✅ | 成本估算: 2小时 |

### 争议点（如有）
| 争议点 | 方案A | 方案B | 决策 | 理由 |
|--------|-------|-------|------|------|
| 详情页方式 | 抽屉 | 模态框 | 抽屉 | 保持上下文，减少页面跳转 |

### 设计工作量
- 页面数: X
- 新组件: X
- 预计前端实现: X 小时
```

## 执行步骤

```
1. [PPQA-1] 自检设计是否符合:
   □ 使用项目已有组件库
   □ 使用项目设计 Token
   □ 无新的视觉语言
   □ 符合现有交互模式
   □ 可实现且成本合理

2. [DAR-1] 如有争议点，进行决策分析
3. [DAR-2] 输出《设计评审报告》
```

## 禁止行为
- 禁止跳过合规检查提交设计
- 禁止有未解决的争议点
