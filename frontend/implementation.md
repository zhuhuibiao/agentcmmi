# 实现 (VER)

## 用途
按设计文档实现页面、组件、API层。

## 触发时机
开发计划表已输出。

## 输入
- 设计文档
- UI设计文档
- 开发计划表

## 执行步骤

### 1. 创建目录结构
```
1. [EXEC-1] 创建工具目录:
   frontend/src/tools/<group>/<tool-key>/
   frontend/tests/<group>/<tool-key>/
```

### 2. 实现 API 调用层
```
2. [EXEC-2] 实现 API 层 (api.ts)
   - 只调用设计文档中列出的API
   - 严格按照设计文档的请求方式调用
   - URL与后端路由完全一致（禁止依赖重定向）
   - 按设计文档组织 Path/Query/Body/Header 参数
```

### 3. 实现类型定义
```
3. [EXEC-3] 实现类型定义 (types.ts)
   - 请求参数类型
   - 响应数据类型
   - 状态类型
```

### 4. 实现组件
```
4. [EXEC-4] 实现页面组件 (components/)
   - 页面标题、字段名称、按钮行为与设计文档一致
   - 页面中的增删改查行为与设计文档一致
   - 页面中的权限表现与设计文档一致
   - 处理状态: 加载中、空状态、成功状态、参数校验错误、接口失败、权限不足
```

### 5. 实现页面状态
```
5. [EXEC-5] 实现页面状态 (hooks/)
   - 状态管理
   - 数据获取
   - 表单处理
```

## API 实现规则

```
1. 必须只调用设计文档中列出的API
2. 必须严格按照设计文档中的请求方式调用API
3. 必须保证前端请求URL与后端实际路由完全一致
   - 包括是否带尾部斜杠
   - 禁止依赖307、308等重定向补正URL
4. 必须严格按照设计文档中的 Path参数、Query参数、Body参数、Header参数 组织请求
5. 设计文档定义统一请求包裹结构时，必须严格按设计文档组装请求体
6. 禁止自行增加 request_id、ts、payload 等设计文档中不存在的包裹字段
7. 必须按设计文档中的成功返回字段读取数据
8. 必须按设计文档中的失败返回字段处理报错
9. 禁止读取设计文档中不存在的响应字段
10. 禁止发送设计文档中不存在的请求字段
```

## PPQA 合规检查（持续进行）

- [PPQA-C1] 类型定义完整 (types.ts)
- [PPQA-C2] Schema 验证 (schemas.ts)
- [PPQA-C3] 组件有清晰命名和注释
- [PPQA-C4] 复用现有组件，不引入新的视觉语言

## 禁止行为
- 禁止伪造设计文档中不存在的API
- 禁止补充设计文档中不存在的业务规则
- 禁止改变当前系统的整体视觉风格
- 禁止修改无关后端文件
- 禁止依赖重定向补正URL
