# 根因分析与决议 (CAR)

## 用途
用于设计缺陷分析、缺陷预防。

## 触发时机
- 前后端实现中发现设计文档与实现不一致
- 测试失败发现设计缺陷
- 用户反馈发现设计问题

## 输入
- 问题描述
- 相关错误日志/证据

## 输出
```markdown
## 缺陷预防报告 (CAR)

### 问题描述
{描述}

### 问题来源
- [ ] 前后端实现中发现
- [ ] 测试失败发现
- [ ] 用户反馈发现

### 5 Whys 分析
- Why 1: {原因}
  → Why 2: {原因}
- Why 2: {原因}
  → Why 3: {原因}
- Why 3: {原因}
  → Why 4: {原因}
- Why 4: {原因}
  → Why 5: {原因}
- Why 5: {根本原因}

### 根本原因分类
- [ ] 技术原因
- [ ] 过程原因
- [ ] 人员原因
- [ ] 工具原因

### 预防措施
- [措施1]: 更新到 {文件} 第{章节}
- [措施2]: {具体内容}

### 涉及的设计文档
- [文档路径]: [更新内容]
```

## 执行步骤

```
1. [CAR-1] 收集问题证据
2. [CAR-2] 复现问题（如可能）
3. [CAR-3] 进行5 Whys分析
4. [CAR-4] 分类根本原因
5. [CAR-5] 制定预防措施
6. [CAR-6] 更新相关设计文档
7. [CAR-7] 输出缺陷预防报告
```

## 5 Whys 分析示例

```
问题: API返回字段与设计文档不一致

Why 1: 为什么API返回字段不一致？
→ 因为后端开发时参考了旧的字段命名

Why 2: 为什么参考了旧的字段命名？
→ 因为没有在开工前核对最新设计文档

Why 3: 为什么没有核对？
→ 因为设计文档存储路径不明确

Why 4: 为什么存储路径不明确？
→ 因为没有统一的文档命名规范

Why 5: 为什么没有统一的文档命名规范？
→ 因为缺少架构规范文档

根本原因: 缺少架构规范文档（过程原因）
```

## 预防措施模板

| 措施类型 | 示例 |
|---------|------|
| 过程改进 | 在开工检查清单中增加"核对设计文档版本" |
| 文档更新 | 更新 api-design-doc.md 增加版本校验要求 |
| 工具支持 | 在 task-list-storage.md 中明确标注文档路径 |

## 禁止行为
- 禁止跳过5 Whys直接给解决方案
- 禁止将症状当原因（如"API返回错误"不是原因）
- 禁止不制定预防措施就关闭问题

## 快速 CAR (Hotfix 场景)

当问题为 Hotfix (已知bug快速修复) 时，可使用简化 CAR:

### 简化流程 (15分钟)

```
F1: 确认问题 (2分钟)
    → 重现步骤是什么？
    → 根本原因是否已知？

F2: 如果原因已知 → 直接修复 + 验证
    如果原因未知 → 走完整 CAR

F3: 记录教训 (5分钟)
    → 写1句话: 这次学到了什么？

F4: 更新知识库 (3分钟)
    → 如果是新类型问题，加入 known-issues.md
```

### 简化模板

```markdown
## Hotfix 教训记录

| 字段 | 内容 |
|------|------|
| 问题 | {problem_one_liner} |
| 根因 | {root_cause} |
| 修复 | {fix_description} |
| 学到 | {lesson_learned} |
| 日期 | {date} |
```
