# 安全评审 (Security Review)

## 概述

本子技能为 Architect 角色提供安全评审能力，在设计阶段识别和缓解安全风险。

**CMMI 过程域：** PPQA (Product Quality Assurance)

**触发时机：** 设计文档完成后，任务派发前。

---

## 1. 安全评审范围

### 1.1 认证与授权
- 用户认证方式（JWT / Session / OAuth）
- 密码存储策略（bcrypt / argon2）
- 权限控制模型（RBAC / ABAC）
- API 访问控制

### 1.2 数据安全
- 敏感数据识别（密码 / Token / PII）
- 数据加密策略（传输加密 / 存储加密）
- 数据库访问控制
- 日志脱敏策略

### 1.3 API 安全
- SQL 注入防护方案
- XSS 防护方案
- CSRF 防护方案
- 输入验证策略
- Rate Limiting

### 1.4 基础设施
- HTTPS 强制
- CORS 配置
- 安全 Headers (CSP / HSTS / X-Frame-Options)
- 密钥管理策略

---

## 2. 安全评审检查清单

```
## 2.1 认证与授权
□ 认证方式已明确（JWT/Session/OAuth）
□ 密码哈希算法已指定（bcrypt/argon2）
□ 权限控制模型已定义（RBAC/ABAC）
□ API 权限矩阵已输出

## 2.2 数据安全
□ 敏感数据字段已标识
□ 加密策略已确定（传输/存储）
□ 日志脱敏规则已定义

## 2.3 API 安全
□ SQL 注入防护措施已设计（参数化查询）
□ XSS 防护措施已设计（输出编码）
□ CSRF 防护措施已设计（Token）
□ 输入验证策略已定义

## 2.4 基础设施
□ HTTPS 配置要求已明确
□ CORS 白名单已定义
□ 安全响应头已规划
□ 环境变量/密钥管理方案已确定
```

## 4. 输出到交接文档

安全评审结果应体现在 `handoff/architect-to-dev.md` 的"安全要求"章节：

```
## 安全设计要求

| 项目 | 要求 | 实现方式 |
|------|------|----------|
| 认证 | JWT + HttpOnly Cookie | 后端实现 |
| 授权 | RBAC | 后端中间件 |
| SQL注入 | 参数化查询 | ORM |
| XSS | 输出编码 | 前端组件 |
```

---

## 5. 禁止行为

- 禁止跳过安全评审直接派发任务
- 禁止将高风险安全问题留到开发阶段
- 禁止未明确认证方式就进入详细设计
